Background

Why Disk Scheduling Matters

When multiple processes request disk access simultaneously, an intelligent queue builds up. The primary cost is seek time — the physical time taken by the read/write head to reach the correct track. Scheduling algorithms minimize this.

📥
I/O Request Queue
Multiple processes simultaneously issue disk access requests, forming a queue of pending track positions that must be ordered intelligently for maximum efficiency.
⏱
Seek Time
The dominant cost of disk access — physical movement of the disk arm between tracks. Minimizing total seek distance directly maximizes system throughput and response time.
📊
Scheduling Goal
OS algorithms determine the optimal order to service requests, each balancing efficiency, fairness, and implementation complexity differently across workload types.
🔄
FCFS Approach
FCFS is the baseline — the simplest possible policy. Process requests in arrival order. No computation needed. Every request guaranteed service. True first-in, first-out fairness.
Algorithm

How FCFS Works

First Come, First Served services requests in the exact order they arrive. Unlike a person in a crowded room calling out requests, FCFS respects the queue — no matter how inefficient the path.

01
Requests Arrive
Disk I/O requests from multiple processes enter a FIFO queue in the order they were issued. Each request specifies a track number between 0 and 199.
02
Head Moves to First
The disk head begins at its current position and moves directly to the first request in the queue. No sorting, no optimization — pure sequential service.
03
Sequential Processing
After servicing each request, the head immediately moves to the next in queue order. This continues until all pending requests are served.
04
Total Seek Calculated
Total head movement = Σ |current − next| for every consecutive pair. Average seek time = total ÷ number of requests.
Real-World Analogy
🎟
Queue at a Ticket Counter
Imagine people lining up at a ticket counter. Person A arrives first, then B, then C. The counter serves them in exactly that order — A, B, C. No skipping, no re-ordering, no priority by proximity. FCFS disk scheduling works identically: the disk head services requests in arrival order, regardless of how far apart the tracks are physically located.
🏧
ATM Queue
An ATM queue enforces strict arrival order. Even if a customer at the back only needs a balance check (a quick operation), they must wait for all earlier customers to complete. Similarly, FCFS may process a distant track request before a nearby one, simply because it arrived earlier.

Fundamentals

Core Concepts

Disk Track
Numbered positions (0–199) on the disk surface where data is stored. Each track forms a concentric circle on the spinning platter.
Disk Head
The read/write mechanism that physically moves between tracks to access stored data. Its movement distance is the primary performance cost.
Seek Time
Time for the head to move from its current track to a requested track. Minimizing this metric is the primary objective of all scheduling algorithms.
Request Queue
The ordered list of track access requests waiting to be serviced. In FCFS, this queue is never reordered — arrival sequence is processing sequence.

Properties

Key Characteristics

  • O(n) complexity: Simply process each request in sequence.
  • Zero starvation: Every request is guaranteed to be served.
  • Fair by design: Arrival order = service order, always.
  • No optimization: Head may swing across the disk repeatedly.

Analysis

Advantages & Disadvantages

Like any algorithm, FCFS involves trade-offs. Understanding them is key to knowing when — and when not — to apply it.

✓ Advantages
🧱
Simplest Implementation
Just a FIFO queue — no sorting, no computation, no decision logic. O(n) time and O(n) space. Any programmer can implement it in minutes.
⚖️
Zero Starvation
Every request is guaranteed to be serviced. No request can be indefinitely postponed because new ones keep arriving at a more favorable position.
🔍
Completely Transparent
The execution order is entirely predictable. Given the arrival sequence, any observer can determine exactly when each request will be serviced.
📐
Great Baseline
Serves as the reference algorithm for benchmarking. Any scheduling improvement must outperform FCFS to justify its added complexity.
✗ Disadvantages
📉
Poor Seek Efficiency
The head can swing from one extreme of the disk to the other repeatedly if requests happen to be far apart in arrival order. Total seek distance can be several times that of SSTF or SCAN.
🐢
High Average Wait Time
Requests at the back of the queue must wait for all preceding requests to complete, even if they are physically close to the current head position. Performance degrades badly under heavy load.
🔁
Not Optimized for Locality
FCFS ignores spatial locality entirely. A cluster of nearby requests may be serviced in the worst possible order if that is how they arrived.
⚡
Not Suitable for High Load
Under heavy I/O workloads, the throughput of FCFS is significantly lower than optimized algorithms. Not appropriate for database servers or busy file systems.
Applications

Real-World Context

Disk scheduling concepts appear across the entire computing stack. Understanding FCFS — and why it is often replaced — provides insight into how real systems are designed.

🖥
Operating Systems
Linux, Windows, macOS — all implement disk I/O schedulers
↓ tap to expand
Linux's I/O scheduler (CFQ/BFQ) moved beyond FCFS decades ago. However, for SSDs with near-zero seek time, the simplicity of FCFS-like NOOP scheduling is perfectly acceptable.
🗄
Database Servers
I/O scheduling is critical for database performance
↓ tap to expand
Database engines like MySQL and PostgreSQL rely heavily on disk I/O. They implement their own buffer pools and write-ahead logging (WAL) to minimize random disk accesses.
📁
File Systems
Ext4, NTFS, APFS optimize disk access patterns
↓ tap to expand
Modern file systems use techniques like block allocation strategies and defragmentation to physically cluster related data together — reducing the effective seek distance.
💾
Storage Controllers
Hardware RAID and SAN controllers manage I/O queues
↓ tap to expand
RAID controllers and Storage Area Networks (SANs) use sophisticated queue-depth management and command reordering — effectively implementing SSTF or SCAN.
☁️
Cloud Storage
AWS EBS, Azure Disks use sophisticated I/O scheduling
↓ tap to expand
Cloud storage services multiplex thousands of I/O requests across physical disks. The scheduling problem becomes multi-dimensional — not just seek distance.
🔬
Research & Education
FCFS as the reference baseline algorithm
↓ tap to expand
FCFS serves as the foundational benchmark in OS research. Any new scheduling algorithm must demonstrate improvement over FCFS to be considered worthwhile.
Summary

Conclusion

FCFS processes disk requests in arrival order — simple, fair, and completely predictable. While it provides zero starvation and minimal implementation overhead, its lack of optimization means the disk head may travel far more than necessary.

Algorithms such as SSTF, SCAN, C-SCAN, and LOOK optimize disk head movement by choosing requests based on proximity or direction of travel. By interacting with the simulations above, we observe how even minor scheduling changes can drastically reduce total seek time.

FCFS remains the foundational algorithm — the baseline every other scheduler is measured against, and the clearest demonstration of the disk scheduling problem itself.
